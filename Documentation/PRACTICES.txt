================================================================================
SOFTWARE ENGINEERING & OBJECT-ORIENTED PROGRAMMING BEST PRACTICES
================================================================================

This document outlines the best practices applied in the Launchpad-FYP project.

================================================================================
GOOD SOFTWARE ENGINEERING (SWE) PRACTICES
================================================================================

1. MODULAR ARCHITECTURE
   - Single Responsibility Principle: Each class has one clear purpose
   - Example: Graph handles token graph, CycleDetector handles cycle detection
   - Benefit: Easy to maintain, test, and extend independently

2. BUILD SYSTEM MANAGEMENT
   - CMake for cross-platform builds (Windows, Linux, macOS support)
   - Automated dependency download (Google Test, nlohmann/json)
   - Configuration-based build flags (Debug/Release)
   - Benefit: Reproducible builds, no manual header path configuration

3. COMPREHENSIVE TESTING
   - Unit tests for all critical components (Token, Pool, Graph, CycleDetector)
   - Google Test framework for industry-standard testing
   - 10/10 test suite passing before release
   - Tests verify edge cases: zero amounts, extreme values, invalid inputs
   - Benefit: Confidence in code correctness, regression prevention

4. CODE COMMENTS & DOCUMENTATION
   - Inline comments explaining complex logic (AMM formula, cycle dedup)
   - Header file documentation describing class responsibilities
   - Clear variable naming (optimalTradeSize, profitUsd, reserveUSD)
   - Algorithm explanations in function headers
   - Benefit: Code readability, knowledge transfer, maintenance ease

5. CONSISTENT NAMING CONVENTIONS
   - camelCase for variables and functions
   - UPPERCASE for constants
   - ClassName for classes
   - prefixWithContext for clarity (profitUsd, amountOutFinal)
   - Benefit: Reduced cognitive load, fewer naming bugs

6. SEPARATION OF CONCERNS
   - Data layer (Token, Pool): Handle core data structures
   - Business logic layer (Graph, CycleDetector, ArbitrageAnalyzer): Processing
   - Presentation layer (HTMLExporter, JSONExporter): Output formatting
   - Benefit: Each layer can be tested and modified independently

7. ERROR HANDLING & VALIDATION
   - Input validation at layer boundaries (nonzero amounts, valid reserves)
   - Descriptive error messages for debugging
   - Named error enum values instead of magic numbers
   - Early return for invalid states
   - Benefit: Easier debugging, clear error propagation

8. VERSION CONTROL PRACTICES
   - .gitignore for build artifacts (build/, node_modules/, artifacts/)
   - Meaningful commit messages describing changes
   - Clean file organization (src/, tests/, contracts/, scripts/, Documentation/)
   - Benefit: Clean repository history, easy collaboration

9. CONFIGURATION MANAGEMENT
   - .env.example template for deployment secrets
   - Environment variables for credentials (PRIVATE_KEY, RPC_URL)
   - No hardcoded sensitive data
   - Benefit: Security, easy local/production switching

10. AUTOMATED DOCUMENTATION
    - Generated reports (HTML, JSON) from analysis results
    - Markdown documentation tracking changes throughout project
    - AI usage log documenting decision rationale
    - Benefit: Audit trail, clear communication, stakeholder visibility

11. CROSS-LANGUAGE INTEGRATION (Part 1 + Part 2)
    - C++ analysis exports JSON
    - TypeScript validation scripts consume JSON
    - Solidity contracts validate on-chain
    - Benefit: Separation of concerns, specialized tools for each domain

12. PERFORMANCE CONSCIOUSNESS
    - Liquidity filtering ($10K minimum) to reduce search space
    - Max depth limit (3) to prevent exponential cycle growth
    - Local reserve copies (no shared state mutation)
    - Efficient data structures (adjacency list, hash maps)
    - Benefit: 474 cycles detected in reasonable time on full dataset

================================================================================
GOOD OBJECT-ORIENTED PROGRAMMING (OOP) PRACTICES
================================================================================

1. SINGLE RESPONSIBILITY PRINCIPLE (SRP)
   Classes: Token, Pool, Graph, CycleDetector, ArbitrageAnalyzer, Exporter classes
   Each handles exactly one aspect of the system
   - Token: Token data (address, symbol, decimals, price)
   - Pool: Pool data + swap calculation (reserves, tokens, fees)
   - Graph: Graph structure (adjacency list, neighbor lookups)
   - CycleDetector: Cycle finding (DFS, canonicalization)
   - ArbitrageAnalyzer: Profitability analysis and ranking
   - Exporters: Output formatting (HTML, JSON)
   Benefit: Classes are compact, focused, and testable

2. ENCAPSULATION
   - Private member variables: pools_, tokenPrice_, graph edges_
   - Public getter methods: id(), reserve0(), reserve1()
   - Private helper methods: escapeHtml(), formatCyclePath()
   - Const correctness: const references for read-only parameters
   - Benefit: Control API surface, prevent invalid state mutations

3. CLEAR INHERITANCE HIERARCHY
   - Base concept: All exporters share interface (export data to format)
   - HTMLExporter and JSONExporter both inherit export pattern
   - Helper methods (escapeHtml, formatPoolIds) logically grouped in HTMLExporter
   - Benefit: Code reuse, consistent interface contract

4. COMPOSITION OVER INHERITANCE
   - ArbitrageOpportunity contains Cycle (not inheriting from it)
   - Pool contains Edge (swap opportunity), not complex inheritance
   - Graph contains Token and Pool objects, not deep class hierarchy
   - Benefit: Flexibility, avoids fragile base class problems

5. CONST CORRECTNESS
   - const methods that don't modify state: Pool::getReserves()
   - const& parameters to avoid unnecessary copies: const Cycle& cycle
   - const vectors for immutable data: const std::vector<ArbitrageOpportunity>&
   - Benefit: Compile-time correctness guarantees, efficient no-copy passing

6. MOVE SEMANTICS & RVALUE REFERENCES
   - std::move() for passing large data efficiently
   - best.amounts = std::move(sim.amounts) (avoids copy)
   - Return value optimization friendly (std::vector returns)
   - Benefit: Performance, avoids unnecessary allocations

7. RESOURCE MANAGEMENT (RAII)
   - Automatic file closing with std::ofstream destructors
   - Vector/map cleanup automatic (no memory leaks)
   - Proper ownership semantics (shared_ptr for pools)
   - Benefit: Exception-safe code, no resource leaks

8. CLEAR INTERFACES
   - Pool::swapAndGetOutput(double amountIn) returns amount
   - Graph::getNeighbors(const std::string& token)
   - CycleDetector::detectCycles(const Graph& graph, int maxDepth)
   - Benefit: Self-documenting code, easy to use correctly

9. CONSTRUCTOR INITIALIZATION
   - Member initialization lists: Pool(const std::string& id, ...) : id_(id)
   - Default values for optional parameters
   - Copy/move constructors explicitly handled (defaulted where appropriate)
   - Benefit: Efficient initialization, clear intent

10. PROPER USE OF STANDARD LIBRARY
    - std::unordered_map for O(1) lookups (tokenUsdPrices_)
    - std::vector for dynamic arrays (cycles, pools)
    - std::set for deduplication (cycleSigs)
    - std::sort with custom comparators (rank by profit)
    - Benefit: Reliable, optimized implementations

11. AVOIDING ANTI-PATTERNS
    - No god objects: Each class has focused responsibility
    - No global state: Data passed as parameters or owned by objects
    - No deep inheritance: Max 1-2 levels in class hierarchy
    - No const_cast or pointer casting (type safe throughout)
    - Benefit: Predictable behavior, easier debugging

12. DOCUMENTATION OF CONTRACTS
    - Class headers explain invariants
    - Function comments describe pre/post conditions
    - Example: "reject if amountIn / reserveIn > 20%" documented in comments
    - Benefit: Clear expectations, easier contract verification

13. EXCEPTION SAFETY
    - Methods provide basic exception guarantee (leaks avoided)
    - File I/O wrapped in try-catch with meaningful errors
    - JSON parsing validates input before loading
    - Benefit: Robust error handling, clean failure modes

14. TEMPLATE USAGE (MINIMAL & APPROPRIATE)
    - std::vector<T>, std::unordered_map<K, V> templated containers
    - No unnecessary template metaprogramming
    - Straightforward generic code that's easy to understand
    - Benefit: Reusability without complexity

15. TESTING DRIVEN BY OOP PRINCIPLES
    - Each class tested independently (TokenTest, PoolTest, GraphTest)
    - Mock data for isolated testing (not using real pools)
    - Tests verify public interface contracts
    - No testing of private implementation details
    - Benefit: Loose coupling, tests remain stable during refactoring
